## Executive Summary

Third-round deep security audit of MitangTriCoreAgent v1.0 focused on six critical areas: event listener lifecycle, memory safety, process safety, data safety, configuration safety, and dependency audit. The prior two rounds' fixes (destroy/removeAllListeners in CoreBus, three-core destroy methods, unhandledRejection handler) were verified as correct. **6 new issues were found and all 6 were fixed**. The overall security posture is strong with zero npm vulnerabilities. The primary concerns addressed were: (1) sensitive config file written without restricted permissions, (2) dangling setTimeout timers in CoreBus trace tracking, (3) a race condition in priority queue flush logic, (4) duplicate SIGINT/SIGTERM handlers across modules, (5) missing uncaughtException handler in standalone mode, and (6) unreleased timeout message tracking set on shutdown.

## Findings

### [F-001] Config file saved without restricted file permissions
- **Category**: OWASP A05 (Security Misconfiguration)
- **Severity**: High
- **Confidence**: 10/10 (actively verified)
- **Location**: `src/config/config-manager.js:106`
- **Description**: `ConfigManager.save()` writes `tricore.json` containing API keys, social tokens, and other sensitive credentials without `mode: 0o600`. On multi-user systems, any local user can read the config file and extract credentials.
- **Exploit Scenario**: An attacker with shell access (but not the app's user) runs `cat data/config/tricore.json` and obtains LLM API keys, Discord/WeChat/Feishu tokens.
- **Reproduction Steps**:
  1. Call `config.save()` (which happens at startup after config migration)
  2. `ls -la data/config/tricore.json` — shows default umask permissions (e.g., 644)
  3. Other users on the system can read the file
- **Remediation**: Added `{ mode: 0o600 }` to `writeFileSync` options and `{ mode: 0o700 }` to `mkdirSync`. Added `chmodSync` best-effort on the config directory.
- **Priority**: P0 (immediate) — **FIXED**

### [F-002] CoreBus `startTrace()` setTimeout never cancelled on destroy
- **Category**: Memory leak / Resource leak
- **Severity**: Medium
- **Confidence**: 9/10 (actively verified)
- **Location**: `src/bus/core-bus.js:325`
- **Description**: Each `startTrace()` call creates a 5-minute `setTimeout` for trace timeout. These timers are never tracked or cancelled in `destroy()`. While `destroy()` clears the `_activeTraces` Map making callbacks into no-ops, the timer handles themselves remain registered with the event loop, accumulating over long-running sessions.
- **Exploit Scenario**: Under heavy load with many concurrent traces, thousands of dangling timers accumulate in the event loop, increasing memory pressure and timer processing overhead.
- **Reproduction Steps**:
  1. Create a CoreBus instance
  2. Call `startTrace()` 1000 times
  3. Call `destroy()`
  4. Check `process.getActiveResourcesInfo()` — timer references remain
- **Remediation**: Added `_traceTimers` Map to track timer handles. `startTrace()` stores timer, `completeTrace()` clears it, `destroy()` cancels all pending timers.
- **Priority**: P1 (this sprint) — **FIXED**

### [F-003] `_flushPriorityQueue` race condition on `_priorityProcessing` flag
- **Category**: Concurrency bug
- **Severity**: Medium
- **Confidence**: 8/10 (code review verified)
- **Location**: `src/bus/core-bus.js:240-248`
- **Description**: When `remaining > 0`, the code sets `_priorityProcessing = false` inside a nested `setImmediate`, but also sets it at the outer level. This creates a scenario where both paths can reset the flag, potentially allowing concurrent flush operations or leaving the flag in an inconsistent state.
- **Exploit Scenario**: Under high event throughput, two concurrent calls to `_flushPriorityQueue()` could both pass the `_priorityProcessing` check, causing events to be delivered out of order.
- **Reproduction Steps**:
  1. Submit >32 NORMAL-priority events rapidly
  2. While the first batch is processing, submit more events
  3. Observe potential flag inconsistency
- **Remediation**: Simplified to set `_priorityProcessing = false` once after processing, then recursively call `_flushPriorityQueue()` directly (not via nested `setImmediate`). Added reset in catch block for robustness.
- **Priority**: P1 (this sprint) — **FIXED**

### [F-004] Multiple SIGINT/SIGTERM handlers cause race on shutdown
- **Category**: Process safety
- **Severity**: Medium
- **Confidence**: 9/10 (actively verified)
- **Location**: Multiple files — `audit-logger.js:105`, `rbac-manager.js:947-948`, `graceful-restart.js:368-369`, `index.js:1830` (formerly)
- **Description**: Four different modules registered SIGINT/SIGTERM handlers via `process.on()`. When a signal arrives, all handlers fire concurrently, potentially calling `agent.stop()` multiple times, writing to closed file handles, or racing on `process.exit()`.
- **Exploit Scenario**: SIGINT triggers: graceful-restart calls `agent.stop()` + `process.exit(0)`, RBAC persists sessions, audit logger flushes + closes, and the standalone entry also calls `agent.stop()`. Concurrent `stop()` calls may corrupt state.
- **Reproduction Steps**:
  1. Start TriCoreAgent in standalone mode
  2. Press Ctrl+C
  3. Observe multiple "stopping" log entries from different handlers
- **Remediation**: 
  - `audit-logger.js`: Changed to `process.once()` and removed `close()`/`exit()` from handler (only flushes)
  - `rbac-manager.js`: Changed to `process.once()`
  - `graceful-restart.js`: Changed to `process.once()`
  - `index.js`: Removed duplicate SIGINT/SIGTERM handlers; graceful-restart handles the full shutdown flow
- **Priority**: P1 (this sprint) — **FIXED**

### [F-005] Missing `uncaughtException` handler in standalone mode
- **Category**: Process safety
- **Severity**: High
- **Confidence**: 10/10 (actively verified)
- **Location**: `src/index.js:1813` (standalone entry point)
- **Description**: The standalone entry point (`if (require.main === module)`) had SIGINT but no `uncaughtException` handler. An uncaught exception would crash the process without cleanup, leaving SQLite databases in an inconsistent state, file handles open, and resources leaked.
- **Exploit Scenario**: A bug in third-party code throws an uncaught exception → Node.js prints stack trace and exits → SQLite WAL files not checkpointed → potential database corruption on next start.
- **Reproduction Steps**:
  1. Start agent in standalone mode
  2. Trigger an uncaught exception (e.g., in a path without try/catch)
  3. Process crashes without cleanup
- **Remediation**: Added `process.on('uncaughtException', ...)` handler that logs the error, calls `agent.stop()` for graceful cleanup, then exits with code 1.
- **Priority**: P0 (immediate) — **FIXED**

### [F-006] `_timedOutMessages` Set not cleaned up in `stop()`
- **Category**: Memory leak (minor)
- **Severity**: Low
- **Confidence**: 8/10 (code review verified)
- **Location**: `src/index.js:845-849`, `stop()` method
- **Description**: The `_timedOutMessages` Set (used to track timed-out message IDs) is lazily initialized and never explicitly cleared in `stop()`. While the agent process exits after `stop()`, this is technically a resource leak in scenarios where `stop()` is called without exiting.
- **Exploit Scenario**: In test suites or restart-without-exit scenarios, the Set accumulates stale message IDs across stop/start cycles.
- **Reproduction Steps**:
  1. Send messages that time out via `sendMessageAsync()`
  2. Call `stop()` then `start()` again
  3. The Set retains old message IDs
- **Remediation**: Added `_timedOutMessages.clear()` and nullification in `stop()` method.
- **Priority**: P2 (next sprint) — **FIXED**
