## Findings

### [F-001] Rate Limit Bypass via X-Forwarded-For Spoofing
- **Category**: OWASP A01 - Broken Access Control
- **Severity**: High
- **Confidence**: 9/10
- **Location**: `src/api/api-server.js:367-373` (original)
- **Description**: `_getRateLimitClientId()` unconditionally trusted the `X-Forwarded-For` header to identify clients for rate limiting. An attacker could send requests with varying `X-Forwarded-For` values to get a fresh token bucket per request, completely bypassing rate limiting. Additionally, the first IP in the chain was used (leftmost), which is the easiest to spoof — a reverse proxy appends the real client IP at the end.
- **Exploit Scenario**: `curl -H "X-Forwarded-For: 1.2.3.$RANDOM" http://target:3721/api/v1/message` — each request gets a new rate limit bucket.
- **Reproduction Steps**: Send 100+ requests with different X-Forwarded-For values; observe all return 200 instead of 429.
- **Remediation**: ✅ **Fixed** — Added `_trustProxy` flag (default `false`), only uses X-Forwarded-For when explicitly enabled. When trusted, uses the last IP in the chain (rightmost, closest to real client).
- **Priority**: P0 (immediate) — Fixed

### [F-002] sendMessageAsync Timeout Resource Leak — Double-Resolution & Post-Timeout Broadcast
- **Category**: OWASP A04 - Insecure Design
- **Severity**: Medium
- **Confidence**: 8/10
- **Location**: `src/index.js:742-808` (original)
- **Description**: `sendMessageAsync()` had no guard against the race condition between the timeout handler and the response handler both calling `resolve()`. If the response arrived at exactly the timeout boundary, both paths could execute, causing: (1) duplicate resolution, (2) post-timeout broadcast of the AI response to the client who already received a timeout error.
- **Exploit Scenario**: Client sends message with 5s timeout. At T=4.999s, the LLM response arrives. Both the response handler and the timeout handler (scheduled for T=5s) execute, sending the client both a success response AND a timeout error, plus duplicate WebSocket broadcasts.
- **Reproduction Steps**: Set a very short timeout (e.g., 1s), send a message that takes ~1s to process. Observe both success and timeout responses.
- **Remediation**: ✅ **Fixed** — Added `settled` boolean guard to prevent double-resolution. Added `_timedOutMessages` Set to track timed-out messages and prevent post-timeout broadcast of AI responses.
- **Priority**: P1 (this sprint) — Fixed

### [F-003] CoreBus Event Information Leakage
- **Category**: STRIDE - Information Disclosure / OWASP A01
- **Severity**: Medium
- **Confidence**: 8/10
- **Location**: `src/index.js:694, 847, 1217` (original)
- **Description**: Three CoreBus event dispatches leaked user-controlled data: (1) `startTrace()` stored `content.substring(0, 50)` — raw user message content in trace context, accessible via `getTrace()`/`getActiveTraces()` APIs; (2) `CONSCIOUSNESS_TASK_REQUEST` dispatched the full `goal` string; (3) `EXECUTION_TASK_FAILED` dispatched the full `error` string which could contain internal stack traces.
- **Exploit Scenario**: An attacker sends a message, then queries the trace API to read other users' message content from the trace context.
- **Reproduction Steps**: Send a message via API, then call `getActiveTraces()` and observe the user's raw content in the trace context.
- **Remediation**: ✅ **Fixed** — (1) Replaced raw content with SHA-256 hash + contentLength; (2) Truncated goal to 50 chars with `goalPreview`; (3) Truncated error messages to 100 chars.
- **Priority**: P1 (this sprint) — Fixed

### [F-004] WebSocket Ping Flood DoS
- **Category**: STRIDE - Denial of Service / OWASP A04
- **Severity**: Medium
- **Confidence**: 8/10
- **Location**: `src/api/api-server.js:686-688, 646` (original)
- **Description**: The WebSocket `ping` message handler (application-level) and protocol-level PING frame handler responded to every request without any rate limiting. A malicious client could send thousands of ping messages per second, consuming server CPU and bandwidth in the processing loop. The `subscribe`/`unsubscribe` handlers were also unlimited — a client could add/remove subscriptions at high frequency.
- **Exploit Scenario**: `while(true) ws.send(JSON.stringify({type:'ping'}))` — server processes each ping, generates pong response, consuming CPU and network resources.
- **Reproduction Steps**: Open a WebSocket connection, send 1000 ping messages in rapid succession, observe server CPU spike.
- **Remediation**: ✅ **Fixed** — Added per-client sliding window rate limiter (`_checkWsClientRateLimit`) with 10 messages/second limit. Messages exceeding the limit are silently dropped (no response, preventing amplification).
- **Priority**: P1 (this sprint) — Fixed

### [F-005] ModelRouter _pendingCalls Memory Leak
- **Category**: OWASP A04 - Insecure Design
- **Severity**: Low
- **Confidence**: 7/10
- **Location**: `src/providers/model-router.js:730-741` (original)
- **Description**: The `_pendingCalls` Map used for request deduplication had no safety timeout. If a provider call hung indefinitely (network issue, unresponsive API), the Map entry would never be cleaned up by the `finally` block, leading to unbounded memory growth. Additionally, subsequent identical requests would receive the hung Promise and also never resolve.
- **Exploit Scenario**: A provider endpoint becomes unresponsive. Over time, 1000 unique requests pile up in `_pendingCalls`, each holding a Promise with closure references, consuming memory that is never reclaimed.
- **Reproduction Steps**: Register a provider that never responds (e.g., point to a non-responding server), send many concurrent requests, observe `_pendingCalls.size` growing without bound.
- **Remediation**: ✅ **Fixed** — Added 5-minute safety timeout timer per pending call. If the call hasn't resolved in 5 minutes, the entry is forcibly removed from `_pendingCalls` and a `dedup_safety_timeout` event is emitted. The timer uses `.unref()` to avoid blocking process exit.
- **Priority**: P2 (next sprint) — Fixed

### [F-006] Missing User Identity Extraction (TODO)
- **Category**: OWASP A04 - Insecure Design / Incomplete Implementation
- **Severity**: Low
- **Confidence**: 10/10 (confirmed by code comment)
- **Location**: `src/core/consciousness-core.js:509` (original TODO comment)
- **Description**: `_injectMemories()` had a TODO comment at line 509: "从消息中提取用户身份" (Extract user identity from messages). Without this, the consciousness core cannot retrieve user-specific memories, leading to generic responses that lack personalization context. The `person` field in the return value was always `null`.
- **Exploit Scenario**: N/A — This is a functionality gap, not a direct vulnerability. However, it means that the `person` field in injected context is always null, and user-specific memory recall never happens.
- **Reproduction Steps**: Call `_injectMemories()` with a message from a known user, observe `person: null` in return value.
- **Remediation**: ✅ **Fixed** — Implemented userId extraction from `message.meta.userId || message.from || message.userId`. Searches for user-specific memories using `用户:${userId}`. Returns a `person` object with `userId`, `relatedMemories` count, and `memorySummaries`. Updated `_buildDynamicContext()` to include `<person-context>` block. Updated the caller in `_processUserMessage()` to pass the `message` object.
- **Priority**: P2 (next sprint) — Fixed

### [F-007] Insufficient Input Validation on API Endpoints
- **Category**: OWASP A03 - Injection / OWASP A01
- **Severity**: Medium
- **Confidence**: 9/10
- **Location**: `src/api/routes/message-routes.js`, `src/api/routes/admin-routes.js`, `src/api/routes/auth-routes.js`
- **Description**: Multiple API endpoints accepted user input without adequate validation: (1) `POST /message` accepted any `content` type and length (no max length, no type check); (2) `POST /activate` accepted unvalidated `provider` names and unlimited `apiKey` length; (3) `POST /admin/restart` accepted unbounded `drainTimeout`/`shutdownTimeout` values; (4) `POST /auth/login` accepted unvalidated `username`/`password` length. These gaps could lead to memory exhaustion (sending huge payloads), type confusion bugs, and injection attacks.
- **Exploit Scenario**: Send a 10MB message via `POST /message`, causing the message queue and LLM context to process an oversized payload, consuming CPU and memory.
- **Reproduction Steps**: `curl -X POST -d '{"content":"'$(python3 -c "print('A'*10000000)")'"}' http://target:3721/api/v1/message` — server processes the oversized content.
- **Remediation**: ✅ **Fixed** — Added comprehensive input validation: (1) `POST /message`: content must be string, max 50K chars, not empty, from max 128 chars, channel whitelist; (2) `POST /activate`: apiKey max 256 chars, provider name regex validation `^[a-zA-Z0-9_-]+$`; (3) `POST /admin/restart`: timeout values clamped to safe ranges; (4) `POST /auth/login`: credential max 256 chars, type validation.
- **Priority**: P1 (this sprint) — Fixed
