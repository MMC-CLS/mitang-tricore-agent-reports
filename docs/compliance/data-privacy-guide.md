# GDPR 数据隐私合规指南 — TriCore Agent v1.0

> **适用范围**: 部署 TriCore Agent 的组织（作为数据控制者）  
> **框架**: EU General Data Protection Regulation (GDPR) 2016/679  
> **最后更新**: 2026-06-11  

---

## 一、数据清单 (Data Inventory)

### 1.1 系统处理的个人数据

TriCore Agent 作为可部署的 AI 智能体系统，其处理个人数据的范围取决于部署方的使用方式。以下是系统**可能**处理的个人数据类别：

| 数据类别 | 具体数据项 | 处理目的 | 存储位置 | 法律依据 |
|---------|-----------|---------|---------|---------|
| **用户账户数据** | 用户名、密码哈希、盐值、角色、IP 地址 | 认证与授权 | SQLite (`rbac_users` 表) + 内存 | 合同履行 (Art.6.1.b) |
| **会话数据** | JWT Token、会话创建时间、IP 地址 | 会话管理 | 内存 + 文件 (`rbac_sessions.json`) | 合法利益 (Art.6.1.f) |
| **审计日志** | 用户 ID、用户名、IP 地址、操作时间戳、操作详情 | 安全合规 | SQLite (`audit_events` 表) + JSON 文件 | 法律义务 (Art.6.1.c) |
| **用户消息内容** | 用户输入的文本、任务目标、上下文数据 | Agent 处理与响应 | 记忆引擎 (SQLite FTS5 + 向量) | 合同履行 (Art.6.1.b) |
| **记忆数据** | 用户对话摘要、执行轨迹、提取的技能 | 智能体持续学习 | SQLite + 向量索引 | 合法利益 (Art.6.1.f) |
| **LLM API 调用** | 提示词（含用户消息和记忆上下文）、模型响应 | AI 推理 | 传输至 LLM Provider（DeepSeek/OpenAI 等） | 合同履行 (Art.6.1.b) |
| **API Key** | 用户生成的 API 密钥 | 编程接口认证 | 内存 + 加密存储 | 合同履行 (Art.6.1.b) |

### 1.2 系统内置的脱敏机制

系统通过以下组件自动识别和脱敏敏感数据：

| 脱敏组件 | 位置 | 覆盖范围 |
|---------|------|---------|
| Content Safety Filter | `src/security/content-safety-filter.js` | PII/代码注入/敏感内容检测 (SAFE/WARN/BLOCK) |
| Audit Logger Redaction | `src/enterprise/audit-logger.js` | 审计日志中自动脱敏 8 类敏感数据 |
| Encryption Service Mask | `src/enterprise/encryption-service.js` | 手机号/邮箱/身份证/银行卡/API Key 结构化掩码 |

### 1.3 数据处理者 (Sub-processors)

| 处理者 | 处理的数据 | 数据处理协议 | 数据驻留 |
|--------|-----------|-------------|---------|
| DeepSeek (默认 Provider) | 提示词内容、系统指令、记忆上下文 | 需签署 DPA | 取决于 API 端点 |
| OpenAI (可选 Provider) | 提示词内容、系统指令、记忆上下文 | [OpenAI DPA](https://openai.com/policies/data-processing-addendum) | 美国/全球 |
| Anthropic (可选 Provider) | 提示词内容、系统指令、记忆上下文 | 需签署 DPA | 美国 |
| 部署方自托管基础设施 | 所有本地数据 | N/A（控制者自有） | 部署方控制 |

## 二、数据保留策略 (Data Retention)

### 2.1 默认保留期限

| 数据类别 | 默认保留期限 | 自动清理机制 | 可配置 |
|---------|------------|------------|--------|
| 活跃会话 | 1 小时 (JWT 过期) | 每分钟定时清理过期会话 | ✅ `sessionTimeout` |
| 审计日志 | 30 天 | 按时间轮转 (`_maxLogAge`) + 按大小轮转 (10MB) | ✅ `maxLogAge` / `maxLogSize` |
| 热记忆 (L0 Hot) | 7 天 (salience ≥ 5) | 每日衰减 (-0.1 salience) | ⚠️ 需代码修改 |
| 温记忆 (L1 Warm) | 30 天 (salience ≥ 3) | 衰减降级 hot→warm→cold | ⚠️ 需代码修改 |
| 冷记忆 (L2 Cold) | 无固定期限 (salience < 3 或 age ≥ 30d) | 超限淘汰 (LRU) | ⚠️ 需代码修改 |
| 执行记忆 (L3 Exec) | 无固定期限 | 技能沉淀源，不自动清理 | ⚠️ 需代码修改 |
| 技能记忆 (L4 Skill) | 永久（除非审计拒绝） | 技能审计 (pending→approved/rejected) | ✅ 审计流程 |
| 加密密钥版本 | 最多 5 个版本 | `_cleanupOldVersions()` 自动清理 | ✅ `maxKeyVersions` |

### 2.2 保留策略实施建议

部署方应：

1. **制定正式的数据保留政策文档** — 明确定义每类数据的保留目的和期限
2. **配置审计日志保留** — 根据业务需求和法律要求设置 `maxLogAge`（建议 90-365 天用于合规）
3. **实施记忆数据的手动清理接口** — 当前记忆衰减是自动的但缺少用户主动删除的 API
4. **LLM Provider 数据保留** — 确认各 Provider 的 API 数据保留策略（OpenAI API 默认不用于训练，但保留 30 天用于滥用监控）

## 三、数据主体访问请求 (DSAR) 程序

### 3.1 当前能力

| 能力 | 状态 | 实现方式 |
|------|------|---------|
| 用户数据查询 | ✅ 可用 | `GET /api/memories?query=...` + `GET /api/audit/logs?userId=...` |
| 用户账户信息导出 | ⚠️ 部分 | `getUsers()` 可查询用户信息，但无标准化导出格式 |
| 完整数据导出 | ❌ 未实现 | 需开发 DSAR 数据打包导出功能 |
| 数据可移植格式 | ❌ 未实现 | 需支持 JSON/CSV 结构化导出 |

### 3.2 DSAR 处理流程（建议实施）

```
数据主体提交请求
      │
      ▼
1. 身份验证 ─── 确认请求者身份（JWT 验证 + 额外身份确认）
      │
      ▼
2. 请求记录 ─── 审计日志记录 DSAR 请求（compliance 级别）
      │
      ▼
3. 数据收集 ─── 跨所有数据存储查询该用户的数据
      │           ├── RBAC 用户表: 账户信息 + 角色
      │           ├── 审计日志: 操作历史
      │           ├── 记忆引擎: 对话记忆 + 执行轨迹
      │           ├── 技能市场: 用户发布的技能
      │           └── 会话数据: 活跃会话
      │
      ▼
4. 数据打包 ─── 结构化 JSON 格式，包含数据来源和保留状态
      │
      ▼
5. 安全交付 ─── 加密传输给数据主体（30 天内响应，GDPR 要求）
      │
      ▼
6. 请求闭环 ─── 审计日志记录交付结果
```

### 3.3 建议实现的 API 端点

```
POST   /api/gdpr/dsar/request     — 提交 DSAR 请求
GET    /api/gdpr/dsar/status/:id   — 查询请求状态
GET    /api/gdpr/dsar/download/:id — 下载数据包（加密）
GET    /api/gdpr/dsar/history      — 查看历史请求（审计员权限）
```

## 四、删除权（被遗忘权）程序

### 4.1 当前能力

| 能力 | 状态 | 实现方式 |
|------|------|---------|
| 用户账户删除 | ⚠️ 部分 | RBAC Manager 有用户管理但无标准删除 API |
| 记忆数据删除 | ⚠️ 部分 | `memory:delete` 权限存在但无批量删除 |
| 审计日志删除 | ❌ 未实现 | 审计日志设计为不可删除（完整性要求） |
| 级联数据删除 | ❌ 未实现 | 无跨存储的用户数据级联清理 |

### 4.2 删除权实施流程（建议）

```
数据主体请求删除
      │
      ▼
1. 身份验证 ─── 确认请求者身份
      │
      ▼
2. 删除范围确认 ─── 明确哪些数据可删除、哪些因法律义务需保留
      │           ├── ✅ 可删除: 记忆数据、技能数据、API Key、会话
      │           ├── ⚠️ 部分删除: 用户账户（匿名化保留审计引用）
      │           └── ❌ 不可删除: 审计日志（法律义务，但可匿名化）
      │
      ▼
3. 执行删除 ─── 按范围逐类删除/匿名化
      │
      ▼
4. 删除验证 ─── 确认所有目标数据已被删除或匿名化
      │
      ▼
5. 删除确认 ─── 向数据主体发送删除确认（30 天内）
```

### 4.3 GDPR Art.17 例外情况

以下数据可免于删除：
- **审计日志** — 法律义务保留（Art.17.3.e），但用户标识可匿名化
- **已发布技能** — 如技能已作为开源内容发布，可能涉及表达自由例外（Art.17.3.a）
- **LLM Provider 端数据** — 需单独向各 Provider 提交删除请求

## 五、数据处理协议 (DPA) 模板参考

### 5.1 DPA 核心条款

部署 TriCore Agent 的组织作为**数据控制者**，应与以下方签署数据处理协议：

#### 与 LLM Provider 的 DPA

```
数据处理协议核心要素:

1. 处理目的: 仅用于提供 LLM API 推理服务，不得用于模型训练（除非明确同意）
2. 数据类别: 文本提示词、系统指令、记忆上下文
3. 数据主体类别: 终端用户（使用 TriCore Agent 的个人）
4. 处理期限: 与 API 服务使用期限一致
5. 子处理者: Provider 的子处理者列表及变更通知机制
6. 技术组织措施 (TOMs):
   - 传输加密 (TLS 1.2+)
   - API Key 认证
   - 访问日志记录
7. 数据泄露通知: 72 小时内通知控制者
8. 数据删除: API 调用完成后 X 天内删除提示词和响应数据
9. 跨境传输: 如涉及，需有充分性决定或标准合同条款 (SCCs)
10. 审计权: 控制者或其指定审计员有权进行合规审计
```

#### 与部署基础设施提供商的 DPA（如使用云服务）

```
如 TriCore Agent 部署在 AWS/Azure/GCP 等云平台：
1. 基础设施层安全控制
2. 静态数据加密（云提供商 KMS）
3. 网络隔离与防火墙
4. 访问控制与 IAM 策略
```

### 5.2 建议签署的 DPA 清单

| 对方 | 角色 | 优先级 |
|------|------|--------|
| LLM Provider (DeepSeek/OpenAI/Anthropic) | 数据处理者 | **P0 — 必须** |
| 云基础设施提供商 (如有) | 次级处理者 | **P0 — 必须** |
| 监控服务提供商 (如有) | 数据处理者 | P1 |
| 第三方审计服务 | 数据处理者 | P2 |

## 六、数据保护影响评估 (DPIA) 触发条件

以下情况需要执行 DPIA（GDPR Art.35）：

| 触发条件 | TriCore Agent 相关性 | 评估 |
|---------|---------------------|------|
| 大规模处理特殊类别数据 | 系统设计不处理特殊类别数据（健康/政治/宗教等），但用户可能输入 | ⚠️ 需评估 |
| 自动化决策（含画像） | Agent 自主决策（任务规划、技能提取）可能构成自动化决策 | ⚠️ 需评估 |
| 系统性监控 | 审计日志和记忆系统持续记录用户行为 | ⚠️ 需评估 |
| 新技术使用 | LLM 和 AI Agent 属于新兴技术 | ✅ 建议执行 |
| 大规模处理个人数据 | 取决于部署规模 | 按部署规模评估 |

**建议**: 在企业部署前执行 DPIA，特别关注自动化决策的透明度和用户干预权。

## 七、数据泄露响应程序

### 7.1 泄露检测

系统内置的异常检测机制：

| 检测机制 | 触发条件 | 通知方式 |
|---------|---------|---------|
| 暴力破解检测 | 5 次/5分钟 失败登录 | `anomaly_detected` 事件 |
| 严重事件激增 | 3 个/1小时 critical 事件 | `critical_surge` 事件 |
| Prometheus 告警 | 7 种告警规则 | Alertmanager (需配置) |

### 7.2 泄露通知时间线

```
泄露发现 (T+0)
    │
    ├── T+1h ─── 内部响应团队激活
    ├── T+24h ─── 影响评估完成
    ├── T+72h ─── 向监管机构通报（GDPR 强制要求 72 小时内）
    └── T+72h+ ── 向受影响数据主体通报（如存在高风险）
```
